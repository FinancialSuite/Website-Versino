Bank Statement Import

⚠ Note: Work in Progress

This module is currently still under active development. Functions, configuration options and the user interface may change at short notice. For productive use, please coordinate with Versino support beforehand.

Overview

The Versino Bank State Import Module enables the import and processing of electronic bank statements in SAP Business One. It supports various bank formats (MT940, CAMT) and uses a powerful rule and RegEx engine to automatically assign transactions and minimize manual work.

Module Access: The process starts under Versino Financial Suite > Bank Statement Import. Processing takes place in the Bank Statement Processing form.

[IMAGE: Bank Statement Processing main form with filled matrix and all relevant columns]

Main Features

Rule-based, Automatic Assignment

The module reads bank statement files and tries to assign transactions automatically. This is done through a multi-stage logic that, among other things, matches bank connections, amounts and reference texts via configurable rules and regular expressions (RegEx).

Interactive Processing Matrix

All transactions are displayed in a clearly arranged matrix. Here you can manually edit unassigned items, assign business partners or manage the status of a row.

Split Function for Collective Postings

If a bank transaction concerns several business cases, you can split it into any number of partial amounts via the Split dialog. The system calculates the remaining amount automatically and creates new, separate rows in the matrix for clean posting.

Status Management

Each row in the matrix has a status. Via the "Cleared" checkbox you can mark a transaction as "cleared" and lock it for further editing. The add-on additionally sets a "Checked by Addon" flag when a row has been processed automatically.

Matrix Columns Overview

The central processing matrix offers the following columns for editing:

Date & Amounts

  • BSLineDate — Posting date of the bank transaction
  • BSValuDate — Value date
  • IncomingAmount — Incoming amounts (credits)
  • OutgoingAmount — Outgoing amounts (debits)

Identification & Codes

  • Ref — Bank reference number
  • Memo / Memo2 — Reference text
  • CardCode — Assigned business partner
  • ExternalCode — External transaction code (bank-specific)
  • InternalCode — Internal transaction code (company-specific)

Status & Processing

  • Cleared — Status checkbox for clearing
  • PstMethod — Posting method
  • BankStatementType — Type of bank transaction

5-Stage Processing Logic

When clicking "Process Statement", each bank transaction passes through five processing stages:

  1. RegEx-based business partner detection — Searching all configured RegEx rules (type "B") and applying to memo fields.
  2. Code transformation via Rule1 — Checking for house bank match (bank code, account number, BIC, IBAN), validating BP bank data, amount check and reference matching.
  3. RegEx-based document detection — Searching all RegEx rules (type "D"), SQL queries for document detection, creation of MultiplePayment entries.
  4. Bank-connection-based BP assignment — Direct match via bank code, account number, BIC and IBAN.
  5. Final RegEx assignment — Use of RegEx results as fallback option.

Status Management

The module manages the following processing states:

  • Open — Not yet processed
  • Assigned — Business partner assigned
  • Cleared — Reconciled with open items (Cleared checkbox set)
  • Split — Divided into several transactions (via split)
  • Completed — Fully processed

The "Checked by Addon" field indicates that a row has been processed automatically by the system:

  • Y — Row was successfully processed
  • (empty) — Row not yet processed

Configuration Tables

The module uses the following user-defined tables for configuration:

  • VAG_BNKSTMNT_R1 — Direct assignment rules (Rule1) with house bank and BP bank data
  • VAG_BNKSTMNT_REGEX — RegEx rules for text detection in memo fields
  • VAG_BNKSTMNT_HEAD / LINES — Import management with FileID, mapping and status tracking
[IMAGE: Configuration table Bank Statement Rule1 with example entries for house bank data and external codes]

Security Note: SQL-Injection Protection

Only SELECT queries are permitted in the RegEx rules. Forbidden SQL operations (UPDATE, INSERT, DELETE, CREATE, DROP, ALTER, TRUNCATE) are blocked automatically by the system.

Split Example

Splitting a collective payment into two items

You receive a collective transfer of 1,000.00 EUR incoming that concerns two business transactions (300 EUR for BP-A, 700 EUR for the original receivable):

  1. Right-click on the 1,000 EUR row → "BP Split"
  2. In the split dialog: enter 300 EUR + business partner BP-A
  3. OK

Result:

  • New row: 300.00 EUR incoming for BP-A
  • Original row: 700.00 EUR incoming (remaining amount)

Integration with SAP Business One

SAP B1 BankStatementsService API

Full integration via the official BankStatementsService API. Direct manipulation of BankStatement objects with transactional safety during split operations.

Business Partner Integration (OCRD)

Direct connection to the OCRD table with Choose-From-List integration, BP validation and status / authorization check.

Application

Setup: Configuring the Automation (for administrators)

The strength of the module lies in its automation engine. It is configured via user-defined tables:

  • Direct Assignment Rules (Rule1): Here you can define fixed rules. Example: If "Rent May" appears in the reference text, the transaction is automatically assigned to a specific external transaction code.
  • RegEx Rules: Here you can store flexible patterns (regular expressions) to extract e.g. invoice or customer numbers from a long reference text and assign them automatically.

Daily Workflow: Processing a Bank Statement

  1. Navigate to Versino Financial Suite > Bank Statement Import and select the bank statement file.
  2. After the import, the Bank Statement Processing form opens. The automation engine has already tried to assign payments.
  3. Click on "Process Statement" to start the 5-stage processing for all rows (or enter a row number to process only this row).
  4. Edit the remaining items manually by assigning the correct business partner.
  5. For collective payments, select the row, activate the Split function via right-click, enter the partial amount and the business partner and confirm.
  6. Mark fully processed rows as "Cleared".
  7. At the end, confirm the entire processing to create the postings.

Tips and Troubleshooting

Tips for Daily Work

  • Use Split consistently: Use the split function for collective payments. A row can only be split if it is not marked as "Cleared".
  • Increase hit rate: Keep your business partner master data (especially bank connections) up to date to increase the automatic hit rate.
  • Test RegEx rules: Test new RegEx patterns thoroughly in an external tool before using them in production.
  • Backup before configuration changes: Create backup copies of the rule configurations before making major changes.
  • Maintenance: Update Rule1 configurations when adding new business partners and review RegEx rules when bank formats change.

Common Problems and Solutions

Problem: "The amount must be greater than 0".

Solution: Enter a valid numeric amount greater than zero in the split dialog.

Problem: "Split not possible, this row is already cleared".

Solution: Remove the "Cleared" checkmark in the row to make it editable again.

Problem: "External transaction code could not be set".

Solution: Check the external code configuration, validate available codes in the system and check user permissions.

Problem: RegEx rules do not work.

Solution: Test RegEx patterns in an external RegEx tool, check SQL queries for syntax errors, validate placeholder substitutions and verify table and field names.

Problem: SQL injection warning on RegEx queries.

Solution: Remove forbidden SQL operations (UPDATE, INSERT, DELETE, CREATE, DROP, ALTER, TRUNCATE). Use only SELECT queries.

Problem: Processing hangs with large data volumes.

Solution: Split the processing into smaller batches, use the specific row processing and optimize RegEx patterns for better performance.
